# Portrait / log split with frozen monthly summaries

Recurring-entity pages (people, channels, projects, topics, services, vendors, campaigns, teams) split into two artifacts:

- **Portrait** at `wiki/<type>/<slug>.md` — holds frontmatter, an **identity zone** (role / areas / collaborators / recurring patterns), open action items, a **monthly chronicle** of frozen one-paragraph summaries, and a "recent activity" list pointing at the latest log files. The canonical `[[<type>/<slug>]]` wikilink always resolves here, so existing references from decisions, incidents, and cross-page links stay valid.
- **Activity log** at `wiki/logs/<type>/<slug>/<YYYY-MM>.md` — one file per calendar month per entity. Append-only dated sections. Empty months are absent; absence is signal. Old months may eventually roll off into `wiki/logs/<type>/<slug>/archive/`, but the monthly summaries on the portrait preserve readable history regardless.

**Ingest owns nearly all portrait writes.** Lint writes exactly two narrow things on portraits — the `## Open action items (from [[todos]])` section (replaced each lint run from `todos.md`) and the `status: dormant` frontmatter flag (the `active → dormant` direction only). Everything else on portraits — identity zone, monthly chronicle, log links, `last_seen`, the `dormant → active` direction, migration — is ingest's. The rationale is timing: ingest runs daily and lint runs weekly, so any synthesis owned by lint would lag the actual entity activity by up to a week. With ingest as the writer for everything time-sensitive, the portrait reflects daily reality.

On each run, ingest does on every touched entity's portrait:

1. **Maintenance** — set `last_seen` to today, flip `dormant → active` if returning, refresh the "Recent activity log" list so the current month's log file is on it. Cheap, mechanical.
2. **Migration if legacy** — if the portrait is still a single-file legacy page (no portrait/log split yet), perform the one-shot split: bucket existing dated sections into monthly log files, synthesise a one-paragraph summary per historic month (marked migration-derived), rewrite the page as a portrait. Bounded to ~5 migrations per ingest run to spread cost; the wiki organically converges over weeks.
3. **Monthly summary fill** — scan the entity's log folder for any month that has a log file but no corresponding frozen summary in the monthly chronicle. For each missing month, read the log file, paraphrase its activity to one paragraph, insert at the top of `## Monthly chronicle` as `### <YYYY-MM>` (newest first). Frozen on insertion — never rewritten.
4. **Identity zone refresh** — only if step 3 added a new monthly summary, re-derive the Role/Areas/Collaborators section from the 3–6 most-recent monthly summaries. If no new summary was added (no new month in the chronicle), leave the identity zone alone — the input hasn't changed.

Three alternatives were rejected. *Single-file with a constantly-rewritten top section* gives the portrait no memory — June erases May. *Single-file with append-only dated sections forever* (the original design) compounds ingest token cost linearly with page age and buries the entity's identity under chronology. *Two-writer split (ingest maintenance + lint synthesis)* leaves the portrait stale for up to a week between lint runs — `last_seen`, the recent-activity list, monthly summaries at month-end, and the dormant→active flip all lag, which is the friction the daily cadence is supposed to remove. Ingest-as-sole-writer keeps daily cadence everywhere on the portrait; lint loses the synthesis role but gains a cleaner janitor role on wiki-wide artifacts.

Cost: the heaviest ingest is the first one of a new calendar month, which writes one summary per touched active entity. With ~10–20 entities touched per ingest, that's ~10–20 small synthesis tasks. Identity zone refresh adds a bounded re-derivation per entity that just gained a summary. Migration is bounded to ~5 entities per run. Day-to-day ingests do maintenance only.
